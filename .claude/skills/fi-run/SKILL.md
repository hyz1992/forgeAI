# 运行时测试 Skill

在代码审查通过后，启动服务进行最终运行时验证。

## 前置条件

- `fi-review` 阶段已完成（代码审查通过）
- 项目代码已生成

## 前置条件验证

在执行前必须依次验证：

1. 确认 `npm install` 或 `pnpm install` 已执行（检查 `node_modules` 是否存在）
2. 确认静态测试已通过（类型检查、单元测试、Lint）
3. 确认代码审查已通过（`docs/review-report.md` 存在且无严重问题）
4. 若验证失败，提示用户先完成 `/fi-review` 阶段

## 执行模式

### 自动修复模式

由 `/fi-auto` 调用时，启用自动修复模式：
- 测试失败时自动调用 `/fi-fix` 修复
- 修复后自动重新运行测试
- 形成闭环直到通过或达到重试限制

### 手动模式

用户直接执行 `/fi-run` 时：
- 测试失败时询问用户是否修复
- 用户确认后调用 `/fi-fix`

## 执行流程

### Step 1: 环境准备

1. 检查并安装 E2E 测试依赖：
   ```bash
   # 如果使用 Playwright
   npx playwright install --with-deps chromium
   ```
2. 检查环境变量配置（`server/.env`、`client/.env`）
3. 确认数据库迁移已执行（如适用）

### Step 2: 启动后端服务

1. 在后台启动后端服务：
   ```bash
   cd server && npm run dev &
   ```
2. 等待服务就绪（检查 `http://localhost:3000/health` 或类似端点）
3. 记录后端进程 PID

### Step 3: 启动前端服务（全栈项目）

1. 在后台启动前端服务：
   ```bash
   cd client && npm run dev &
   ```
2. 等待服务就绪（检查 `http://localhost:5173` 或配置的端口）
3. 记录前端进程 PID

### Step 4: 服务健康检查

验证服务正常运行：

| 检查项 | 验证方式 |
|--------|----------|
| 后端 API 响应 | `curl http://localhost:3000/health` |
| 前端页面加载 | `curl http://localhost:5173` |
| CORS 配置 | 跨域请求测试 |
| 数据库连接 | 执行简单查询 |

**若健康检查失败**：
- 输出错误日志
- 尝试自动修复（自动模式）或询问用户（手动模式）
- 修复后重新检查

### Step 5: E2E 测试（含自动修复循环）

运行 Playwright E2E 测试：

```bash
npx playwright test
```

**测试范围**：
- 关键用户流程（登录、核心操作、退出）
- 页面导航
- 表单提交
- 错误处理

**失败处理流程**：

```
E2E 测试失败
    ↓
保存截图到 test-results/
    ↓
分析失败原因（超时、元素未找到、断言失败等）
    ↓
【自动模式】直接调用 /fi-fix 修复
【手动模式】询问用户是否修复
    ↓
/fi-fix 修复代码
    ↓
重新运行 E2E 测试
    ↓
通过 → 继续 | 失败 → 重试（最多 3 次）
```

**重试限制**：
- 单个测试用例最多重试 3 次
- 总修复次数最多 5 次
- 超过限制后暂停，提示用户手动介入

### Step 6: UI 截图分析（可选）

使用 MCP 工具进行 UI 视觉验证：

1. **截取关键页面截图**：
   - 首页
   - 主要功能页面
   - 表单页面

2. **使用 MCP 分析截图**：
   ```
   mcp__4_5v_mcp__analyze_image
   - 检查布局是否正确
   - 检查关键元素是否可见
   - 检查响应式设计（可选）
   ```

3. **输出分析报告**

**若发现 UI 问题**：
- 记录问题详情
- 自动模式：调用 `/fi-fix` 修复
- 手动模式：询问用户是否修复

### Step 7: 清理与报告

1. 停止后台服务进程
2. 生成运行时测试报告：

```markdown
# 运行时测试报告

## 服务启动

| 服务 | 状态 | 地址 | 启动时间 |
|------|------|------|---------|
| 后端 | ✅ 运行中 | http://localhost:3000 | 2.3s |
| 前端 | ✅ 运行中 | http://localhost:5173 | 1.8s |

## 健康检查

- [x] 后端 API 响应正常
- [x] 前端页面可访问
- [x] 数据库连接正常

## E2E 测试

| 测试用例 | 状态 | 耗时 |
|---------|------|------|
| 用户登录流程 | ✅ 通过 | 1.2s |
| 核心功能操作 | ✅ 通过 | 3.5s |
| 表单提交验证 | ✅ 通过 | 0.8s |

## 自动修复记录

| 轮次 | 问题 | 修复方式 | 结果 |
|------|------|---------|------|
| 1 | 登录按钮未找到 | 调整选择器 | ✅ 通过 |
| 2 | 表单提交超时 | 增加等待时间 | ✅ 通过 |

## UI 分析

- 布局：正常
- 关键元素：全部可见
- 响应式：正常

## 统计

- 总测试用例：XX
- 通过：XX
- 自动修复次数：XX
- 总耗时：XX 分钟
```

### Step 8: Git 提交

测试通过后，自动提交到 git（如果项目是 git 仓库）：

```bash
git add . && \
git commit -m "docs: 添加运行时测试报告

- E2E 测试：XX/XX 通过
- 服务健康检查：通过
- UI 验证：通过

ForgeAI 自动提交 - $(date '+%Y-%m-%d %H:%M:%S')"
```

## 输出

- 运行时测试报告（`docs/runtime-test-report.md`）
- E2E 测试截图（`test-results/`）
- UI 分析结果

## 错误处理

| 错误类型 | 自动模式 | 手动模式 |
|---------|---------|---------|
| 服务启动失败 | 尝试修复配置 | 提示检查端口/配置 |
| 健康检查失败 | 自动修复 | 询问是否修复 |
| E2E 测试失败 | 自动修复循环 | 询问是否修复 |
| UI 分析发现问题 | 自动修复 | 询问是否修复 |
| 重试超限 | 暂停，提示手动介入 | 暂停，展示问题列表 |

## 超时设置

| 操作 | 超时时间 |
|------|---------|
| 服务启动 | 30 秒 |
| 健康检查 | 10 秒 |
| E2E 单个测试 | 30 秒 |
| E2E 全部测试 | 5 分钟 |
| UI 截图分析 | 2 分钟 |
| 单次修复 | 3 分钟 |

## 与其他 Skill 的关系

```
/fi-fix → 静态测试通过
    ↓
/fi-review → 代码审查通过
    ↓
/fi-run → 运行时测试（本阶段）
    │
    ├── 测试通过 → 完成 ✅
    │
    └── 测试失败 → /fi-fix 修复 → 重新测试
                   ↑______________|
                   （自动修复循环，最多 3 次）
```
