# 需求收集 Skill

用户**无需手写需求文档**，只需提供产品/项目的粗略想法，通过对话即可自动生成 `docs/requirements.md`。

## 前置条件

- 无（这是整个流程的入口）

## 功能说明

将用户碎片化或粗略的想法，通过结构化对话整理成完整的需求文档。

**核心理念**：
- 用户只需提供想法，AI 负责整理
- 一次只问一个问题，降低认知负担（来自 brainstorming）
- 提供选项加速对话，也允许自由输入
- 动态调整问题，根据项目类型定制

## 执行流程

### Step 0: 检查并安装外部依赖

直接运行安装脚本，它会自动检测并安装缺失的依赖：

```bash
python .claude/scripts/install.py
```

脚本会自动检查 superpowers plugin 等依赖，已安装的会跳过，未安装的会提示安装。

### Step 1: 调用 Brainstorming

调用 `@superpowers:brainstorming` 开始需求探索：

```
@superpowers:brainstorming

目标：收集项目需求，生成 docs/requirements.md

探索重点：
1. 项目目标（一句话说明）
2. 核心功能（必须有什么）
3. 非功能需求（性能、安全、技术约束）
4. 边界条件（不做什么）
5. 验收标准（怎么算完成）
```

### Step 2-8: 对话收集

按以下顺序收集信息：
1. **初步询问**：请用一两句话描述你想做什么
2. **项目类型**：全栈/后端/前端/CLI/类库/其他
3. **技术栈选型**：根据项目类型推荐技术栈，用户可接受或自定义
   - **全栈 Web 项目**（推荐）：
     - 后端：Node.js + Fastify
     - 前端：Vue 3 + Vite + TypeScript
     - 数据库：SQLite + Prisma（保留升级 PostgreSQL 的可能）
   - **纯后端**：Node.js + Fastify + SQLite/PostgreSQL + Prisma
   - **纯前端**：Vue 3 + Vite + TypeScript + Pinia
   - **CLI 工具**：Node.js + TypeScript + Commander/Inquirer
   - **类库**：Node.js + TypeScript + Vitest
   - 用户可选择"使用推荐"或"自定义技术栈"
4. **核心功能**：循环收集直到用户说"没有了"
5. **AI 服务**：如果涉及 AI，确认需要哪些能力
   - 需要哪些 AI 能力？（LLM/生图/TTS/视频）
   - API Key 如何管理？（环境变量/配置文件/密钥管理服务）
   - 是否需要 Mock 支持？（开发/测试环境）
6. **外部服务**：是否需要集成其他外部服务？
   - 支付服务（支付宝/微信支付/Stripe）
   - 地图服务（高德/百度/Google Maps）
   - 第三方 API（其他）
   - 认证服务（OAuth/SSO）
7. **复杂交互**：是否涉及复杂 UI 交互？
   - 多步骤表单/向导
   - 拖拽/画布交互
   - 实时协作（WebSocket）
8. **横切关注点**：
   - 权限控制需求（RBAC/ABAC/简单角色）
   - 审计日志需求
   - 缓存策略需求
9. **技术约束**：本地部署/高并发/移动端/离线等
10. **目标用户**：个人/小团队/中型企业/大型企业
11. **汇总确认**：展示需求汇总，等待确认

### Step 9: 生成文档

用户确认后，生成 `docs/requirements.md`（若 `docs/` 目录不存在则先创建）。生成时须按 `.claude/shared-references/requirements-structure.md` 所定义的结构与必填项输出，确保含项目名称、目标、类型，至少一个核心功能（含描述、优先级、验收标准），以及至少一条整体验收标准，以便通过后续 `validate.py requirement` 校验。

### Step 10: 自动校验并完成

生成 `docs/requirements.md` 后，**必须自动执行**（无需用户手动运行）：

```bash
python .claude/scripts/validate.py requirement
```

- **校验通过**：继续下一步
- **校验失败**：根据脚本输出的缺失项（如缺少项目名称、目标、核心功能、验收标准）补充或修正 `docs/requirements.md`，再次运行上述校验命令；若多次仍不通过，提示用户补全后重新执行 `/fi-init`。

### Step 11: Git 提交

校验通过后，自动提交到 git（如果项目是 git 仓库）：

```bash
# 检查是否是 git 仓库
git rev-parse --is-inside-work-tree 2>/dev/null && \
# 检查是否有更改
git status --porcelain && \
# 提交更改
git add . && \
git commit -m "docs: 完成需求收集

- 生成 docs/requirements.md
- 项目类型：{项目类型}
- 技术栈：{技术栈简介}
- 核心功能：{功能数量} 个

ForgeAI 自动提交 - $(date '+%Y-%m-%d %H:%M:%S')"
```

完成后提示：
```
✅ 需求文档已生成并校验通过：docs/requirements.md
✅ 已提交到 git

下一步：/fi-plan 开始架构设计
```

## 询问策略

### 问题优先级

1. **必问**：项目目标、项目类型、技术栈、核心功能
2. **按需问**：AI 服务（如果涉及 AI）、外部服务、复杂交互
3. **可推断**：验收标准、边界条件（可从功能推断）

### 交互原则

- **一次一问**：不要一次问多个问题
- **提供选项**：a/b/c 加速选择
- **允许跳过**：用户可以说"不清楚"、"跳过"
- **智能推断**：根据已有信息推断后续答案
- **随时可改**：确认前都可以修改

### 处理碎片化输入

当用户的初步想法过于简短或模糊时，使用以下策略挖掘有效信息：

**1. 5W1H 引导法**

按需追问，帮助用户理清思路：

| 维度 | 示例问题 |
|------|----------|
| **Who** | 这个项目是给谁用的？个人/团队/企业？ |
| **What** | 核心要解决什么问题？用户最痛的点是什么？ |
| **Why** | 为什么想做这个？现有方案哪里不满足？ |
| **When** | 有时间要求吗？需要什么时候上线？ |
| **Where** | 在哪里运行？Web/小程序/桌面/移动端？ |
| **How** | 有偏好的技术或实现方式吗？ |

**2. 场景化提问**

用具体场景帮助用户思考：
- "能描述一下用户最典型的使用流程吗？"
- "用户打开这个应用后，第一件事会做什么？"
- "如果用户想完成 XX 任务，需要几步操作？"

**3. 关键词展开**

从用户描述中提取关键词并展开追问：
- 用户说"类似 XX" → "具体哪些功能要像 XX？哪些不需要？"
- 用户说"管理" → "需要管理什么内容？谁有权限管理？"
- 用户说"自动" → "自动到什么程度？需要人工干预吗？"
- 用户说"分析" → "分析什么数据？输出什么结果？"

**4. 示例启发**

提供同类产品的功能示例帮助用户确认：
- "类似的产品有 A、B、C，你的想法更接近哪个？"
- "功能 A 需要吗？功能 B 呢？"

**5. 反向确认**

通过问"不做什么"来明确边界：
- "有什么是明确不想做的吗？"
- "第一版不需要考虑什么功能？"

**处理流程**：

```
用户输入碎片化 → 识别缺失信息 → 选择合适的追问策略 → 逐个补充 → 汇总确认
```

## 与其他 Skill 的关系

```
/fi-init → 生成 docs/requirements.md
  ↓
/fi-plan → 读取 docs/requirements.md，生成架构设计
  ↓
... 后续流程
```

## 错误处理

| 情况 | 处理方式 |
|------|----------|
| 用户输入模糊 | 提供选项让用户选择 |
| 用户中途退出 | 保存已收集信息，下次可恢复 |
| 需求过于复杂 | 建议拆分为多个子项目 |
